# ypTrend 技术方案（MVP v1.6，双轨落地版）

## 1. 目标与范围
- 目标：围绕 AI 大模型与 AI 编程方向，提供稳定的热点发现、判断与输出能力。
- 范围：当前同时维护两条能力链路。
  - Web 平台链路：`server + web`，用于持续监控、飞书通知、SSE 实时展示。
  - 自包含 Agent Skill 链路：`skills/hot-monitor`，无需本地后端即可采集与生成报告。

## 2. 架构总览

### 2.1 Web 平台链路（在线监控）
- 前端：React + Vite + Tailwind CSS v4 + shadcn + Aceternity UI
- 后端：Node.js + Express
- 存储：SQLite（better-sqlite3）
- 调度：node-cron
- AI 判定：OpenRouter（无 Key 时启发式降级）
- 通知：飞书 Webhook（首行关键词校验 `trend`）
- 实时推送：SSE（`GET /api/stream`）

### 2.2 自包含 Skill 链路（离线可移植）
- 目录：`skills/hot-monitor`
- 运行方式：Python 脚本直接调用外部公开源/API，不依赖 `server` 进程。
- 输出：结构化 JSON 中间结果 + Markdown 热点报告。
- 分析：由 Agent 根据 `references/analysis-guide.md` 做相关性/真实性/重要性判断。

## 3. Web 平台链路技术细节

### 3.1 处理流程
1. 读取启用规则（`monitors.enabled=1`）。
2. 多源采集候选（Twitter/Web/RSS）。
3. URL 归一化 + `unique_hash` 去重。
4. 主题聚类与跨源佐证（`crossSourceCount`）。
5. 7 天新鲜度窗口过滤。
6. AI 判定（`isRelevant/isCredible/shouldNotify/confidence/reason`）。
7. 来源门槛拦截（Twitter 与 Web/RSS 分别判定）。
8. 飞书通知落库。
9. SSE 推送 `hot_item`/`notification` 到前端。

### 3.2 评分与门槛
- Twitter 互动分：
  - `engagementScore = 1.5*likes + 2*retweets + 2*quotes + replies + views/100`
  - 原创过滤：reply/quote/retweet 直接过滤。
  - 通知门槛：`engagement_score >= twitterQuality.notifyMinEngagementScore`（默认 2000）。
- Web/RSS 信号分：
  - `signalScore = sourceAuthority + recency + corroboration + contentQuality + aiConfidenceBonus`
  - 高信号门槛：`signalTier = high` 且阈值由 `nonTwitterSignal.highThreshold`（默认 70）控制。
- 全来源共同前置条件：
  - `isRelevant = true`
  - `isCredible = true`
  - `shouldNotify = true`
  - 新鲜度窗口内
  - 未通知去重

### 3.3 数据与接口
- 关键表：`monitors`、`hot_items`、`ai_evaluations`、`notifications`、`settings`
- `hot_items` 扩展字段：`engagement_score`、`view_count`、`engagement_tier`、`signal_score`、`signal_tier`、`trust_level`
- 核心 API：
  - `GET /api/health`
  - `GET/POST/PATCH/DELETE /api/keywords`
  - `GET /api/hot-items`
  - `GET /api/notifications`
  - `GET/PUT /api/settings`
  - `POST /api/run-now`
  - `GET /api/stream`

## 4. 自包含 Skill 设计（hot-monitor）

### 4.1 目录结构
```text
skills/hot-monitor/
  SKILL.md
  agents/openai.yaml
  references/
    search-sources.md
    analysis-guide.md
  scripts/
    search_web.py
    search_china.py
    search_twitter.py
    generate_report.py
```

### 4.2 脚本职责
- `search_web.py`
  - 采集 Google News RSS、Bing News RSS、HN Algolia、DuckDuckGo HTML。
- `search_china.py`
  - 采集中文公开热点源（知乎/B站/微博），按关键词做基础过滤。
- `search_twitter.py`
  - 调用 `twitterapi.io`，需要 `TWITTERAPI_IO_KEY`。
  - 无 Key 自动跳过，不中断流程。
- `generate_report.py`
  - 合并多源 JSON、去重、按时间排序并输出 Markdown 报告模板。

### 4.3 分析框架
参考 `references/analysis-guide.md`，按四维评分：
- Relevance（相关性）
- Credibility（可信度）
- Timeliness（时效性）
- Impact（影响面）

总分映射到 `P0/P1/P2/Drop`，并在结论中显式标注证据来源与不确定性。

### 4.4 降级策略
- 单源失败不影响整体流程，记录 `source_status` 并继续。
- Twitter Key 缺失时返回 `skipped` 状态。
- 保证报告可生成，即使候选为空也输出标准模板。

## 5. 方案边界与兼容策略
- Web 平台链路继续服务在线监控、通知、实时看板。
