# ypTrend 技术方案（MVP v1.5，已实现对齐版）

## 1. 架构概览
- 前端：React + Vite + Tailwind CSS v4 + shadcn + Aceternity UI。
- 后端：Node.js + Express。
- 存储：SQLite（better-sqlite3）。
- 定时任务：node-cron。
- AI 判定：OpenRouter（无 Key 时启发式降级，含限流兜底）。
- 通知：飞书 Webhook（首行关键词校验：`trend`）。
- 实时推送：SSE（`/api/stream`）。

## 2. 当前实现状态（2026-04-16）
1. 主链路已打通
- 采集 -> 去重 -> AI 判定 -> 通知判定 -> 飞书通知 -> SSE 推送 -> 前端实时刷新。

2. 采集源
- Twitter：`twitterapi.io`。
- Web：DuckDuckGo + Bing Web RSS + Bing News RSS。
- RSS：Google News RSS（`SEARCH_LOCALES` 多语言）。

3. 稳定性
- 新鲜度窗口清理、去重、失败兜底、可复跑验收脚本已具备。

## 3. 后端处理流程
1. 拉取启用监控规则（`monitors.enabled = 1`）。
2. 按来源采集候选（支持单源 run-now 和全源轮询）。
3. 候选标准化与去重（URL 归一化 + `unique_hash`）。
4. 主题聚类与跨源佐证计算（`crossSourceCount`）。
5. 新鲜度窗口过滤（默认 7 天）。
6. AI 判定（`isRelevant` / `isCredible` / `shouldNotify` / `confidence` / `reason`）。
7. 按来源执行统一硬门槛拦截。
8. 命中后发送飞书并写通知流水。
9. 通过 SSE 推送 `hot_item` 与 `notification` 事件到前端。

## 4. 评分模型与门槛策略

### 4.1 Twitter 质量与推送策略
1. 原创过滤（严格）
- reply / quote / retweet 直接过滤。

2. 互动分公式
- `engagementScore = 1.5*likes + 2*retweets + 2*quotes + replies + views/100`
- 浏览量缺失按 `views=0` 处理。

3. 互动等级（用于展示）
- `low < 500`
- `medium = 500-1999`
- `high >= 2000`

4. 推送门槛（用于通知拦截）
- 采用可配置阈值：`twitterQuality.notifyMinEngagementScore`（默认 2000）。
- 判定逻辑：`engagement_score >= notifyMinEngagementScore` 才允许进入通知阶段。

### 4.2 Web/RSS 可比评分模型
- `signalScore = sourceAuthority + recency + corroboration + contentQuality + aiConfidenceBonus`
- 范围：`0-100`

1. `sourceAuthority`（0-35）
- `rss/google_news_rss = 35`
- `web/bing_news = 32`
- `web/bing_web = 24`
- `web/duckduckgo = 18`

2. `recency`（0-25）
- `<= 6h: 25`
- `<= 24h: 20`
- `<= 72h: 12`
- `<= 7d: 4`
- `> 7d: 0`

3. `corroboration`（0-25）
- `crossSourceCount = 1: 0`
- `crossSourceCount = 2: 15`
- `crossSourceCount >= 3: 25`

4. `contentQuality`（0-5）
- 标题 + 有效摘要：`5`
- 仅标题：`2`
- 其他：`0`

5. `aiConfidenceBonus`（0-10）
- `round(confidence * 10)`

6. `signalTier`
- `low < 50`
- `medium = 50-69`
- `high >= threshold`（threshold 默认 70，可配置）

### 4.3 统一通知硬门槛
- Twitter：满足 `engagement_score >= twitterQuality.notifyMinEngagementScore`。
- Web/RSS：满足 `signalTier = high`（由 `nonTwitterSignal.highThreshold` 决定）。
- 全来源共同前置条件：
  - `isRelevant = true`
  - `isCredible = true`
  - `shouldNotify = true`
  - 7 天窗口内
  - 去重（未发送过）

### 4.4 跨源可靠性策略
- `reliability.lowTrustRequireCrossSource = true` 时，低可信候选需满足 `crossSourceCount >= minDistinctSources`。

## 6. API 与实时事件

### 6.1 主要 API
- `GET /api/health`
- `GET/POST/PATCH/DELETE /api/keywords`
- `GET /api/hot-items`
- `GET /api/notifications`
- `GET/PUT /api/settings`
- `POST /api/run-now`

### 6.2 输出增强
1. `GET /api/hot-items`
- 返回结构化质量字段（互动分、信号分、等级、可信等级）。

2. `GET /api/notifications`
- 返回通知基础字段 + 关联热点字段 + 最新 AI 判定字段：
  - `ai_is_relevant`
  - `ai_is_credible`
  - `ai_confidence`
  - `ai_reason`
  - `ai_model`
  - `ai_should_notify`

### 6.3 SSE
- 连接端点：`GET /api/stream`
- 事件类型：
  - `connected`
  - `hot_item`
  - `notification`
- 前端打开页面后可实时接收新热点与新通知，无需刷新。
