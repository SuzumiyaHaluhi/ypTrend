## 今日修复记录（2026-04-19）

### 1. OpenRouter 无 Key 时误通知风险修复
- 问题：无 `OPENROUTER_API_KEY` 场景下，系统存在“相关/可信默认放行”倾向，可能误触发通知。
- 处理方法：统一改为启发式评估结果，并在无 Key 时强制 `shouldNotify=false`。
- 处理逻辑：保留热点入池与可见性，但关闭自动通知，避免在降级路径下制造高成本误报。

### 2. 设置保存被 Webhook 校验阻塞问题修复
- 问题：用户仅调整扫描参数时，也会因为空 `feishuWebhook` 不满足 URL 校验而保存失败。
- 处理方法：后端 schema 允许空字符串或合法 URL；前端提交前对 webhook 执行 `trim` 并改进错误提示提取。
- 处理逻辑：将“通知通道是否已配置”与“系统参数是否可保存”解耦，确保配置体验稳定。

### 3. Web/RSS 采集静默失败问题修复
- 问题：`Promise.allSettled` 仅吞错并返回空结果时，无法区分“真实无结果”与“上游异常”。
- 处理方法：为 Web/RSS 增加 provider 失败聚合；部分失败时附带降级告警；全量失败时抛出结构化错误。
- 处理逻辑：让采集失败可观测、可追踪、可在前端状态中显式反馈，避免误判系统健康状态。

### 4. Twitter Retweet 配置未生效修复
- 问题：`excludeRetweets` 配置存在但未真正参与过滤决策。
- 处理方法：在质量评估中按 `excludeRetweets` 控制 retweet 是否过滤。
- 处理逻辑：恢复“配置即行为”的一致性，避免出现参数可配但无效的隐性缺陷。

### 5. 默认持久化策略修复
- 问题：默认 `DB_FILE=:memory:` 与平台长期使用预期不一致，重启后数据丢失风险高。
- 处理方法：默认数据库改为落盘 `data/yptrend.db`，并同步 README 说明 `DB_FILE` 覆盖方式。
- 处理逻辑：提升开箱即用可靠性，同时保留显式切换内存库的能力用于临时/测试场景。

### 6. 验收脚本与新策略对齐
- 问题：旧版 `accept:p0-1` 假设无 Key 场景仍会产生通知，已不符合新安全策略。
- 处理方法：更新验收断言为“无 Key 场景必须无通知，但应有热点与系统告警事件”。
- 处理逻辑：保证验收标准与产品策略一致，避免“代码正确但验收失败”的伪回归。
